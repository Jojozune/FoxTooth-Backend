Account Management

POST
/account/create
Create a new player account with email and password
Request
{ "display_name": "PlayerName", "player_tag": "#0001", "email": "player@example.com", "password": "SecurePass123" }
Validation Rules:
  • display_name: 2-20 chars, alphanumeric + underscores
  • player_tag: # + 3-6 uppercase alphanumeric
  • email: valid email format
  • password: 8+ chars, uppercase, lowercase, digit
Success Response (201)
{ "status": "success", "player_id": 42, "message": "Account created successfully. Please login." }
Error Response (400/409)
{ "status": "error", "message": "The combination PlayerName#0001 is already taken" }
POST
/account/login
Authenticate player and receive JWT access/refresh tokens
Request
{ "display_name": "PlayerName", "player_tag": "#0001", "password": "SecurePass123", "remember_me": true }
Success Response (200)
{ "status": "success", "player_id": 42, "session_code": "ABC123", "token": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...", "refresh_token": "a1b2c3d4e5f6...", "remember_token": "x9y8z7w6v5u4...", "token_expires_in": "2 hours", "server": { "ip": "192.168.1.100", "port": 7777 } }
‚ö†Ô∏è Note: The remember_token is only included if remember_me: true. Store this securely in Keychain/Vault for passwordless login.
POST
/account/connect
Reconnect existing player without password
Request
{ "display_name": "PlayerName", "player_tag": "#0001", "remember_me": false }
Success Response (200)
{ "status": "success", "player_id": 42, "session_code": "ABC123", "token": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...", "refresh_token": "a1b2c3d4e5f6..." }

Authentication & Tokens

GET
/player/validate-token
REQUIRES AUTH
Validate current JWT token
Headers:
Authorization: Bearer eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...
Success Response (200)
{ "status": "valid", "player": { "playerId": 42, "displayName": "PlayerName", "playerTag": "#0001" }, "message": "Token is valid" }
POST
/player/refresh-token
Exchange refresh token for new access token
Request
{ "refresh_token": "a1b2c3d4e5f6...", "player_id": 42 }
Success Response (200)
{ "status": "success", "token": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...", "token_expires_in": "2 hours" }
POST
/player/remember-login
Exchange remember token for access + refresh tokens (passwordless)
Request
{ "player_id": 42, "remember_token": "x9y8z7w6v5u4..." }
Success Response (200)
{ "status": "success", "token": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...", "refresh_token": "a1b2c3d4e5f6...", "token_expires_in": "2 hours" }
POST
/player/logout
REQUIRES AUTH
Logout player and invalidate sessions
Parameters (optional):
  • remember_token: Logout this device only (if provided)
  • No body: Logout globally (all devices)
Success Response (200)
{ "status": "success", "message": "Logged out successfully" }

Error Codes & Responses

Common Error Codes

Code Status Message Cause
400 Bad Request Invalid input or missing fields Validation failed on request body
401 Unauthorized Invalid or expired token Missing/invalid JWT token in header
403 Forbidden Action not allowed Insufficient permissions
404 Not Found Resource not found Player, session, or invite doesn't exist
409 Conflict Duplicate or conflicting data Player tag already exists, duplicate invite
429 Too Many Requests Rate limit exceeded Too many requests from same IP
500 Server Error Internal server error Database or server issue

Standard Error Response Format

{ "status": "error", "message": "Detailed error message", "code": 400, "details": { "field": "password", "reason": "Password must contain uppercase and lowercase letters" } }

API Usage Best Practices

‚úÖ DO:

  • Always send Authorization: Bearer {token} header for authenticated endpoints
  • Refresh tokens when they expire (before 2 hours)
  • Implement heartbeat every 30 seconds to keep player alive
  • Handle all error codes gracefully with user-friendly messages
  • Use HTTPS in production for all API calls
  • Store refresh and remember tokens securely (never in localStorage)

‚ùå DON'T:

  • Expose tokens in URL parameters or logs
  • Send sensitive data in query strings
  • Cache authentication tokens longer than their expiry
  • Ignore rate limits - implement exponential backoff for retries
  • Store plain passwords - use tokens exclusively
  • Make assumptions about player online status without heartbeat
